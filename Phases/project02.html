Phase Description

Pretty pictures are not very pretty if they cannot be seen. So this phase is all about the GUI. To start, you should download this JAR file. It contains a single class, FractalPanel. FractalPanel is a subclass of JPanel that adds 2 methods which, when combined, display a fractal. The first method is updateImage(). This method takes in your 2-d array of int with the calculated escape time for each pixel. This is the fractal that will then be drawn in the panel. It does this by using the value that was previously provided by a second method: setIndexColorModel(). This second method specifies the color scheme that is used to draw the fractal. It takes in an instance of IndexColorModel which defines a set of colors. For each entry in the array, the FractalPanel uses the escape time to find the color at that index in the IndexColorModel. That color is used to draw that pixel.The details of the FractalPanel class are documented here .

Everyone has a different idea of what colors make the fractals most attractive. Rather than force my (objectively better) design choices, each group must define 4 different color models for users to choose from. To get you started you can (but do not have to) use this class which has 3 methods which create color models. Even with this, you will need to create a color model on your own. My only request is that groups do not create color schemes that blind the user or give them a migraine.

We also need something which will let the user choose from the four fractal algorithms and four color schemes should be used. Give your program a menubar with at least 3 menus : File, Fractal, and Color. The File menu should contain an option to quit the program, the Fractal menu should contain menu items for each fractal, and the Color menu should contain menu items for each of your color schemes. When the Exit item is selected, your program should quit. When a fractal is selected, you should calculate the fractal and display it using the most recently selected color scheme. When a color scheme is selected, you should set it so that fractals will be displayed using this color scheme until a different choice is made or the program exits.

Finally, the last phase relied on an escape distance of 2. Provide the user with some method of changing to a new escape distance. Your program must either prevent users from entering an invalid escape distance OR show an error when the escape distance they enter is not a number greater then 0. If the user provides an escape distance greater than 0, recalculate the last chosen fractal and have the FractalPanel display the recalculated escape times.

Remember, hours of coding makes up for minutes of planning.

How to Submit

Phase #2 can only be submitted by the team member whose username comes first alphabetically. Submission by other students will not be accepted nor can they be used to show that work was completed before the submission deadline. We strongly encourage each team schedule a meeting which includes all members to do the Web-CAT submission. Any students not included in the list of submission group members will receive a 0.

The project submission must contain the entire Eclipse project including the properly Javadoc'd source code. This includes both the implementation code AND the JUnit test code.

Phase #2 Grading Rubric

The team grades for phase #2 will use the following rubric:

  • Each team must have a file named run.txt which tells the TAs how to run your program to verify you have a working GUI [1 point]
  • Each team defines a GUI application that includes a FractalPanel instance to see the fractal images [2 points]
  • Each team's program must include a menu bar with the required menus and menu items [2 points]
  • Each team's program exits the program when the exit menu item is selected [2 points]
  • Each team's program must display the appropriate fractal when its menu item is selected [4 * 4 = 16 points]
  • Once a color scheme is selected, the program uses that color scheme to display all fractals [4 * 4 = 16 points]
  • Each team include a file named update.txt which tells the TAs how to update the escape distance and verify invalid times are rejected/impossible. [1 points]
  • Each team's program does not allow a user to enter an invalid escape distance or shows an error message and rejects the invalid escape distance [2 points]
  • Each team's program accepts valid escape distance and recalculates and redraws the fractal when given a new escape distance [4 points]
  • Once a new escape distance is entered, the program continues to use that time when generating other fractals [2 points]
  • Each team must write JUnit tests showing that their project can: [50 points]
    • JUnit tests from phase #1 have been updated (where this is needed) so they continue to pass [26 points]
    • When the escape distance is set to 3, calculates the escape time for a coordinate whose distance from the origin exceeds the escape distance after a 10 passes of the loop [4 * 6 points = 24 points]
      • Mandelbrot Set: (0.46007827788650374, -0.3383561643835661)
      • Julia Set: 1.4538160469667272, -0.13502935420743645)
      • Burning Ship Set: (-1.6999999999999802, 0.0030136986301371603)
      • Multibrot Set: (0.7025440313111545, -0.5520547945205528)
  • All classes, methods, and instance variables are commented using Javadoc [2 points]
